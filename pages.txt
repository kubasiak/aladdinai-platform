AUTHENTICATION FLOWS
================================================================================

New Customer Registration (by provider):
    1. Provider logs in → /provider.html
    2. Provider clicks "Add Customer" tab
    3. Provider enters email/username + password
    4. Customer account created in Firebase Auth
    5. Customer document created in Firestore
    6. Provider sees success message

Customer First Login:
    1. Customer goes to /login.html
    2. Customer enters email/username + password
    3. Firebase Auth validates credentials
    4. System checks if customer has template selected
    5a. If no template → redirect to /template-selector.html
    5b. If has template → redirect to /admin.html

Customer Editing Page:
    1. Customer logged in at /admin.html
    2. Customer edits content (contenteditable fields)
    3. Customer uploads media (videos, audio)
    4. Customer generates screenshots from videos
    5. Customer clicks "💾 Save All Changes"
    6. Settings saved to Firestore
    7. Static HTML generated and saved to Storage
    8. Public page updated at /:slug

Customer Viewing Public Page:
    1. Anyone visits /:slug (e.g., /aladdinai)
    2. Cloud Function receives request
    3. Function loads public-pages/{slug}.html from Storage
    4. Static HTML served to browser
    5. No Firestore queries, instant load

Provider Deleting Page:
    1. Provider logs in → /provider.html
    2. Provider clicks "📄 Delete Page" for customer
    3. Confirmation dialog appears
    4. Provider confirms deletion
    5. Settings and templateType deleted from Firestore
    6. Static page removed from Storage
    7. Media files remain (videos, audio, screenshots)
    8. Customer account remains active
    9. Customer must select template again

Provider Deleting Account:
    1. Provider logs in → /provider.html
    2. Provider clicks "🗑️ Delete Account" for customer
    3. Double confirmation dialog appears
    4. Provider confirms deletion
    5. Customer auth account deleted
    6. Firestore customer document deleted
    7. All media files deleted from Storage
    8. Static page removed
    9. Customer completely removed from system

================================================================================
REDIRECT RULES (Firebase Hosting)
================================================================================

Priority Order:
1. Static files served first (if file exists in /public)
2. / → servePage Cloud Function
3. /:slug → servePage Cloud Function

Examples:
    /admin.html → Serves static file (not Cloud Function)
    /login.html → Serves static file (not Cloud Function)
    / → Cloud Function → public-pages/index.html
    /aladdinai → Cloud Function → public-pages/aladdinai.html
    /index.html → Serves static redirect page

================================================================================
ERRORS & 404s
================================================================================

404 Scenarios:
    1. /:slug not found
        - User visits non-existent slug
        - Cloud Function returns custom 404 HTML
        - Message: "No page found for: {slug}"

    2. Static file not found
        - Firebase Hosting returns standard 404
        - Redirects handled by firebase.json
